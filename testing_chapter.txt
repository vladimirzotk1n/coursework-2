ГЛАВА. ТЕСТИРОВАНИЕ
===================


1. МЕТОДИКА ПРОВЕДЕНИЯ ТЕСТИРОВАНИЯ
-------------------------------------

Основной целью тестирования являлась проверка корректности взаимодействия всех компонентов системы: REST API, базы данных PostgreSQL, объектного хранилища MinIO и фонового процесса. Особое внимание уделялось проверке конкурентного доступа, транзакционной целостности данных, корректности работы триггеров PostgreSQL и механизма хранения файлов.

Тестирование выполнялось против реальных экземпляров PostgreSQL и MinIO, автоматически разворачиваемых в контейнерах Docker. Такой подход позволил запускать проверки в окружении, максимально приближенном к рабочему. После запуска контейнеров автоматически применялась актуальная схема базы данных, что исключало расхождения между тестовой и рабочей конфигурацией системы. Проверка HTTP-интерфейса осуществлялась через выполнение запросов непосредственно к приложению FastAPI.

Каждая проверка выполнялась в полностью изолированной среде: перед каждым тестом все таблицы базы данных очищались каскадным удалением, что гарантировало отсутствие побочных эффектов между независимыми тестами.


2. ПОДГОТОВКА ДАННЫХ
---------------------

Для наполнения базы данных демонстрационными данными был разработан скрипт scripts/seed.py. Он предназначен для использования в среде разработки и запускается одной командой внутри контейнера:

    docker compose run --rm backend python -m scripts.seed

По умолчанию создаётся 50 пользователей. Флаг --users N позволяет задать произвольное количество:

    docker compose run --rm backend python -m scripts.seed --users 10

При повторном запуске скрипт проверяет наличие уже созданных пользователей и пропускает генерацию, если данные уже существуют. Флаг --reset позволяет предварительно удалить всех ранее созданных тестовых пользователей (удаление каскадно распространяется на связанные эксперименты, запуски, серии, точки, отчёты и файлы).

Для воспроизводимости результатов генератор случайных чисел инициализируется фиксированным значением (seed = 42), что гарантирует одинаковый набор данных при каждом запуске.

2.1. Состав создаваемых пользователей

Скрипт создаёт пятьдесят пользователей, разделённых на два типа.

Десять именных пользователей имеют фиксированные, заранее известные учётные данные:

  — demo  (demo@example.com)           пароль: demo123
  — alice (alice.johnson@gmail.com)    пароль: alice2024
  — bob   (b.smith42@yahoo.com)        пароль: physics42
  — carol (carol.w@hotmail.com)        пароль: waveform!
  — dave  (dave.k@outlook.com)         пароль: thermo99
  — eva   (eva.petrov@gmail.com)       пароль: optics25
  — frank (f.miller@mail.ru)           пароль: circuit7
  — grace (grace.chen88@gmail.com)     пароль: quantumG8
  — henry (h.taylor@yandex.ru)         пароль: pendulum1
  — ivan  (ivan.sokolov@mail.ru)       пароль: measure!2

Остальные сорок пользователей генерируются процедурно из словарей имён и фамилий (русских и международных), случайных почтовых провайдеров (gmail.com, yahoo.com, outlook.com, mail.ru, yandex.ru и других) и уникальных паролей физической тематики вида photon42!, gauss71!, planck92@ и т.п. Имена пользователей формируются по шаблону first_last (например, aleksei_smirnov, darya_jones). Генерация детерминирована: при одном и том же значении seed результат идентичен.

Пароли всех пользователей хешируются параллельно в пуле потоков (до восьми потоков одновременно), поскольку bcrypt является вычислительно интенсивной операцией. Это существенно сокращает время подготовки данных по сравнению с последовательным хешированием.

2.2. Состав и структура экспериментов

Для каждого пользователя создаётся полный набор из десяти экспериментов, описывающих реальные физические опыты. Порядок следования экспериментов у каждого пользователя перемешивается случайно. Перечень экспериментов:

  1. Затухающие колебания математического маятника — моделирует затухающий гармонический процесс (модель: A·exp(−γt)·cos(ωt)).
  2. Проверка закона Ома для металлического проводника — линейная зависимость I(U) (k = 0,42 А/В).
  3. Разряд конденсатора через резистор (RC-цепь, C = 100 мкФ, R = 10 кОм) — экспоненциальный спад U(t).
  4. Свободное падение тела — квадратичная зависимость координаты от времени.
  5. Дифракция Фраунгофера на одной щели — распределение интенсивности по закону sinc².
  6. Тепловое расширение алюминиевого стержня (L₀ = 500 мм) — линейная зависимость удлинения от температуры.
  7. Зависимость периода маятника от его длины — проверка соотношения T = 2π√(L/g).
  8. Закон Бойля-Мариотта — изотермический процесс, обратная зависимость давления от объёма.
  9. Резонансная кривая RLC-контура — зависимость амплитуды напряжения от частоты (кривая Лоренца).
  10. Калибровка термопары типа K в диапазоне 0–200 °C — линейная зависимость ЭДС от температуры.

2.3. Генерация запусков, серий и точек

Для каждого эксперимента создаётся от трёх до пяти запусков. Для каждого запуска генерируются одна или две серии данных. Точки в серии равномерно распределяются вдоль оси x в заданном диапазоне, к теоретическому значению добавляется случайный шум по нормальному закону. Каждая точка содержит порядковый номер, значения x и y, а также погрешности по обеим осям. Количество точек в серии определяется шаблоном эксперимента:

  — от 15 до 25 точек (зависимость периода маятника от длины);
  — от 20 до 40 точек (закон Бойля-Мариотта);
  — от 30 до 50 точек (расширение стержня, калибровка термопары);
  — от 40 до 60 точек (закон Ома, свободное падение, резонанс RLC);
  — от 60 до 80 точек (разряд конденсатора);
  — от 60 до 100 точек (затухающие колебания);
  — от 80 до 120 точек (дифракция Фраунгофера).

2.4. Генерация файловых вложений

К каждому запуску с вероятностью 20 % прикрепляется изображение (PNG), схематично воспроизводящее подпись эксперимента и номер запуска. С вероятностью 50 % создаётся отчёт, включающий:

  — исходный файл в формате LaTeX (.tex);
  — итоговый файл в формате PDF, сгенерированный через matplotlib (имитация страницы A4);
  — от одного до трёх фотографий-вложений (PNG) в виде синтетических точечных графиков.

Для 30 % серий автоматически генерируется PNG-график с планками погрешностей (средствами matplotlib), который сохраняется как вложение серии.

Все файлы загружаются в объектное хранилище MinIO по установленной схеме путей:

  — изображения запуска:   images/{run_id}/{file_id}.png
  — исходник отчёта:       reports/{report_id}/report.tex
  — PDF отчёта:            reports/{report_id}/report.pdf
  — вложения отчёта:       reports/{report_id}/{file_id}.png
  — графики серий:         plots/{series_id}/{file_id}.png

Загрузка файлов выполняется в два шага: сначала в таблицу Files вставляется временная запись с путём-заглушкой, затем путь обновляется до финального значения, содержащего file_id, и файл передаётся в хранилище.

2.5. Итоговый объём данных

Итоговый набор данных для пятидесяти пользователей в типичном запуске:

  — пользователей:    50  (10 именных + 40 сгенерированных)
  — экспериментов:    500 (10 на каждого)
  — запусков:         ~2 000 (в среднем 4 на эксперимент)
  — серий данных:     ~3 200
  — точек измерений:  ~90 000–100 000
  — графиков (PNG):   ~960
  — изображений:      ~400
  — отчётов:          ~1 000
  — фото-вложений:    ~1 500


3. СЦЕНАРИИ ТЕСТИРОВАНИЯ И РЕЗУЛЬТАТЫ
--------------------------------------

Проверки разбиты на восемь групп по функциональным областям и охватывают HTTP-интерфейс, триггеры базы данных, взаимодействие с объектным хранилищем и работу фонового процесса.


3.1. Регистрация пользователей и проверка токенов (Группы 1–2)

Проверка 1.
Регистрация нового пользователя с корректными данными (имя, адрес почты, пароль).
Результат: ответ с кодом 201; в теле содержатся поля username, email, user_id; поле password_hash отсутствует.

Проверка 2.
Повторная регистрация с именем пользователя, которое уже занято (адрес почты отличается).
Результат: ответ с кодом 409.

Проверка 3.
Повторная регистрация с адресом почты, который уже занят (имя пользователя отличается).
Результат: ответ с кодом 409.

Проверка 4.
Регистрация с паролем из трёх символов (меньше установленного минимума).
Результат: ответ с кодом 422.

Проверка 5.
Вход в систему с корректными учётными данными ранее зарегистрированного пользователя.
Результат: ответ с кодом 200; в теле содержится access_token длиной более 20 символов и token_type со значением "bearer".

Проверка 6.
Вход с верным именем пользователя, но неверным паролем.
Результат: ответ с кодом 401.

Проверка 7.
Вход с именем пользователя, которого нет в системе.
Результат: ответ с кодом 401.

Проверка 8.
Запрос к защищённому ресурсу с корректным токеном.
Результат: ответ с кодом 200.

Проверка 9.
Запрос к защищённому ресурсу без заголовка авторизации.
Результат: ответ с кодом 401.

Проверка 10.
Запрос к защищённому ресурсу с заведомо недействительным токеном.
Результат: ответ с кодом 401.

Проверка 11.
Запрос к защищённому ресурсу с заголовком авторизации, использующим нераспознаваемую схему (Token вместо Bearer).
Результат: ответ с кодом 401.


3.2. Управление экспериментами (Группы 3–5)

Проверка 1.
Создание эксперимента с полями названия и описания.
Результат: ответ с кодом 201; в теле присутствуют название, описание, идентификатор эксперимента, временны́е метки создания и последнего изменения.

Проверка 2.
Запрос списка экспериментов от пользователя, не создавшего ни одного.
Результат: ответ с кодом 200 и пустым массивом.

Проверка 3.
Два пользователя создают по одному эксперименту, после чего каждый запрашивает свой список.
Результат: каждый пользователь получает ровно одну запись — только свой эксперимент. Изоляция данных между пользователями соблюдается.

Проверка 4.
Запрос существующего эксперимента по его идентификатору.
Результат: ответ с кодом 200; идентификатор в ответе совпадает с запрошенным.

Проверка 5.
Запрос эксперимента с несуществующим идентификатором.
Результат: ответ с кодом 404.

Проверка 6.
Пользователь B запрашивает эксперимент, созданный пользователем A.
Результат: ответ с кодом 404. Чужие данные недоступны.

Проверка 7.
Изменение названия и описания существующего эксперимента.
Результат: ответ с кодом 200; оба поля обновлены в соответствии с переданными значениями.

Проверка 8.
Изменение эксперимента — проверка работы триггера trg_updated_at_experiments.
Результат: поле updated_at в ответе не меньше значения, зафиксированного в момент создания. Триггер обновил временну́ю метку.

Проверка 9.
Пользователь B пытается изменить эксперимент пользователя A.
Результат: ответ с кодом 404.

Проверка 10.
Удаление существующего эксперимента.
Результат: ответ с кодом 204; последующий запрос по тому же идентификатору возвращает код 404.

Проверка 11.
Удаление эксперимента, к которому привязаны запуск, серия данных и точка измерения.
Результат: после удаления запросы к ранее существовавшим запуску и серии возвращают код 404. Каскадное удаление средствами PostgreSQL (ON DELETE CASCADE) сработало по всей иерархии: Experiments → ExperimentRuns → DataSeries → DataPoints.


3.3. Управление запусками экспериментов (Группы 3–4)

Проверка 1.
Создание первого запуска в эксперименте.
Результат: ответ с кодом 201; поле run_number равно 1; поля name и experiment_id совпадают с переданными.

Проверка 2.
Последовательное создание трёх запусков в одном эксперименте.
Результат: поле run_number принимает значения 1, 2, 3 строго по порядку — триггер trg_run_number работает корректно.

Проверка 3.
По одному запуску создаётся в двух разных экспериментах.
Результат: в обоих экспериментах run_number равно 1. Счётчик порядкового номера независим для каждого эксперимента.

Проверка 4.
Восемь одновременных запросов на создание запуска в одном эксперименте (через asyncio.gather).
Результат: все восемь ответов содержат код 201; значения run_number образуют непрерывный ряд от 1 до 8 без повторений. Механизм pg_advisory_xact_lock в функции fn_assign_run_number сериализовал конкурентные вставки и предотвратил конфликты.

Проверка 5.
Создание трёх запусков, затем запрос их списка.
Результат: ответ с кодом 200; значения run_number в списке отсортированы по возрастанию.

Проверка 6.
Изменение имени и описания существующего запуска.
Результат: ответ с кодом 200; оба поля обновлены.

Проверка 7.
Изменение запуска — проверка работы триггера updated_at.
Результат: поле updated_at в ответе не меньше исходного значения.

Проверка 8.
Удаление существующего запуска.
Результат: ответ с кодом 204; последующий запрос по тому же идентификатору возвращает код 404.

Проверка 9.
Пользователь B запрашивает запуск, принадлежащий пользователю A.
Результат: ответ с кодом 404.


3.4. Серии данных и точки измерений (Группы 3–5)

Проверка 1.
Создание серии данных с указанием имени, единицы измерения по оси x и по оси y.
Результат: ответ с кодом 201; все переданные поля и идентификатор запуска присутствуют в ответе с корректными значениями.

Проверка 2.
Добавление точки измерения (порядковый номер, значения x и y, погрешности), затем запрос списка точек.
Результат: создание возвращает код 201 и правильное значение y; список содержит одну точку с корректным порядковым номером.

Проверка 3.
Три точки добавляются в порядке 3 → 1 → 2, затем запрашивается список.
Результат: точки возвращаются в порядке 1, 2, 3 — данные сортируются по полю measurement_order.

Проверка 4.
Добавляется одна точка, затем массовая замена двумя новыми точками.
Результат: ответ с кодом 200; список содержит ровно две точки с новыми значениями — исходная точка удалена.

Проверка 5.
Попытка массовой замены точками, у двух из которых одинаковый порядковый номер.
Результат: ответ с кодом 422 — проверка данных на уровне API отклонила дублирование порядкового номера.

Проверка 6.
Изменение имени серии и единиц измерения.
Результат: ответ с кодом 200; имя серии обновлено.

Проверка 7.
Удаление серии, содержащей одну точку измерения.
Результат: запрос удалённой серии возвращает код 404; прямой запрос к таблице DataPoints по идентификатору серии возвращает ноль строк — каскадное удаление сработало.

Проверка 8.
Пользователь B запрашивает серию, принадлежащую пользователю A.
Результат: ответ с кодом 404.


3.5. Загрузка и удаление файлов (Группа 6, частично Группа 8)

Проверка 1.
Загрузка PNG-изображения для запуска.
Результат: ответ с кодом 201; в теле содержатся идентификатор файла, тип содержимого image/png, размер в байтах, соответствующий размеру переданного файла, и путь в хранилище вида images/{run_id}/…; объект фактически существует в MinIO по указанному пути.

Проверка 2.
Загрузка двух изображений, затем запрос их списка.
Результат: ответ с кодом 200; список содержит два элемента, каждый из которых включает URL-ссылку, начинающуюся с http (подписанная ссылка для прямого доступа).

Проверка 3.
Попытка загрузить файл формата PDF вместо изображения.
Результат: ответ с кодом 415. Сервер отклоняет недопустимый тип содержимого.

Проверка 4.
Попытка удалить изображение с несуществующим идентификатором.
Результат: ответ с кодом 404.

Проверка 5.
Загрузка изображения, затем его удаление. Прямая проверка состояния базы данных.
Результат: запрос к таблице Files по идентификатору файла возвращает пустой результат. Триггер fn_cleanup_orphaned_file удалил запись из Files после того, как была удалена последняя ссылка на файл из таблицы RunImages.

Проверка 6.
Загрузка изображения, затем его удаление. Проверка постановки задания в очередь.
Результат: в таблице FileDeletionQueue появляется запись с соответствующим путём и незаполненным полем processed_at. Триггер fn_file_outbox добавил задание на удаление из хранилища, которое ожидает обработки фоновым процессом.


3.6. Фоновый процесс удаления файлов (Группы 7–8)

Проверка 1. Полный жизненный цикл файла.
Загружается изображение — объект существует в MinIO. Затем изображение удаляется через API: строка из RunImages удаляется, триггер удаляет запись из Files, следующий триггер создаёт ожидающую запись в FileDeletionQueue; объект в MinIO на этом этапе ещё присутствует. После этого вызывается функция обработки фонового процесса process_batch.
Результат: функция вернула значение не менее 1; поле processed_at записи в очереди заполнено, поле last_error пусто; объект в MinIO больше не существует.

Проверка 2.
Функция обработки вызывается дважды подряд для одной уже завершённой записи.
Результат: значение processed_at после второго вызова совпадает со значением после первого — повторная обработка не производится.

Проверка 3. Полный каскад при удалении эксперимента.
Создаётся эксперимент с запуском, к запуску прикрепляется изображение. Затем удаляется сам эксперимент.
Результат: запись из Files удалена триггером fn_cleanup_orphaned_file; в FileDeletionQueue создана ожидающая запись. После вызова process_batch объект удалён из MinIO, поле processed_at заполнено. Полная цепочка каскадного удаления сработала корректно: Experiments → ExperimentRuns → RunImages → Files → FileDeletionQueue → MinIO.

Проверка 4.
Вручную вставляется запись FileDeletionQueue с путём, которого нет в MinIO. Вызывается process_batch.
Результат: поскольку операция удаления объекта из S3-совместимого хранилища идемпотентна (удаление несуществующего ключа не является ошибкой), запись помечается как обработанная. В случае если библиотека всё же выбрасывает исключение, фоновый процесс увеличивает счётчик попыток — запись не теряется.
